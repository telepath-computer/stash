# Watch

## Why

We agreed on a "slick + magical" autosync experience, but some behavior is still underspecified:

- `stash watch install` lifecycle on macOS (preflight, idempotency, failure modes)
- how foreground/background watch output should work
- cross-process safety when multiple syncs run for the same stash
- default `stash sync` output (single-line progress instead of per-file stdout spam)
- a minimal, predictable `stash watch status`

Without explicit rules, implementation will drift and user experience will feel inconsistent.

## What changes

### Scope split stays the same

- `stash.md` remains core Stash behavior (methods + sync algorithm)
- `cli.md` owns command UX, watch orchestration, service management, logs, and status output

CLI lifecycle/watch events remain CLI-owned (not added to Stash core events). Stash continues exposing `mutation` for sync actions.

## Target: `spec/cli.md`

### 1. `stash watch install` / `uninstall` / `status` semantics

#### `stash watch install`

Define exact behavior:

1. macOS-only command
2. preflight resolve `stash` executable from `PATH`
3. if `stash` is not resolvable: fail with hint:
   - "Couldn't find `stash` on PATH"
   - "From `code/`, run `npm link`"
4. ensure directories exist:
   - `~/Library/LaunchAgents`
   - `~/.stash`
5. write plist:
   - `~/Library/LaunchAgents/com.stash.watch.plist`
   - runs `stash watch --all`
   - `RunAtLoad = true`
   - `KeepAlive = true`
   - stdout/stderr -> `~/.stash/watch.log`
6. if already loaded, unload best-effort then reload
7. start immediately (or rely on `RunAtLoad`)
8. print one success line

Idempotency:

- rerun is safe
- if unchanged and healthy: no-op success
- if changed: rewrite plist + reload

#### `stash watch uninstall`

- unload job best-effort
- remove plist
- keep log file
- idempotent if already uninstalled

#### `stash watch status` (minimal v1)

Output only:

- `watch service: running|not running`
- `registered stashes: N`
- optional `last error: ...` (if available)

Exit codes:

- `0` if running
- `1` if not running

### 2. Watch triggers + defaults

Keep and make explicit defaults:

- filesystem change trigger with debounce = `1000ms`
- periodic poll = `30s`
- `stash watch --all` re-reads `~/.stash/stashes.json` on each poll tick to pick up new/removed paths

No registry-file watcher in v1 (poll-based refresh only).

### 3. Cross-process sync lock behavior

Define a per-stash lock used by all sync entry points:

- lock file: `.stash/sync.lock`
- acquired via atomic create (`wx`)
- includes metadata (`pid`, `startedAt`, `hostname`, `runId`)
- stale lock TTL = `10 minutes`
- stale/dead owner can be reclaimed (remove + retry once)
- release in `finally`

Policy:

- manual `stash sync` when lock held -> fail fast, non-zero exit
- watch-triggered sync when lock held -> skip cycle and retry on next trigger/poll

This complements in-process single-flight by adding cross-process protection.

### 4. `stash sync` output UX

Replace default mutation-per-line stdout with compact progress UX:

- TTY mode:
  - one live-updating line with spinner + elapsed time + phase/progress
  - one final summary line on success
  - one final error line on failure
- non-TTY mode:
  - no spinner
  - concise start/end summary only
- `--verbose`:
  - enables detailed mutation lines (old style)

Running-line intent:

- feels "alive" (elapsed time + phase + counters)
- never floods stdout by default

### 5. Watch/service logs

Define file and format:

- path: `~/.stash/watch.log`
- format: JSON Lines (one JSON object per line)
- include timestamp, level, event, stash path (when applicable), and message
- never log secrets/tokens

v1 size policy:

- if file exceeds `10MB`, truncate/reset to empty before appending new events
- no rotation scheme yet

### 6. Watch overlap model (keep simple)

No ownership lease/coordination system in v1.

- `stash watch` (single path) and `stash watch --all` may coexist
- safety is guaranteed by per-stash sync lock
- duplicate watchers are tolerated; concurrent sync on same stash is prevented by lock

## Target: `spec/stash.md`

### 1. Mention cross-process lock in `stash.sync()` behavior

In the `sync()` behavior/flow, add that sync acquires a per-stash lock (`.stash/sync.lock`) before running and releases it in `finally`.

This should be framed as core safety behavior, independent of whether sync was called manually or by watch.

### 2. Keep lifecycle event ownership unchanged

No new `sync:start/done/error` events are added to `StashEvents` in this change. CLI/watch lifecycle remains defined in `cli.md`.

## Notes

- This changeset is intentionally UX/spec-level only.
- No implementation details beyond behavior contracts and defaults.
