# Watch

## Why

Stash syncs are manual — you run `stash sync` each time. The goal is a "set and forget" experience: start `stash watch`, and your stash stays in sync automatically. Local edits are pushed, remote changes are pulled, merges happen silently in the background.

This changeset adds the `stash watch` command, updates `stash sync` output to use a shared visual language, and adds a cross-process sync lock to `Stash.sync()`.

## Behavior

### Sync output

Both `stash sync` and `stash watch` share the same visual language for sync progress. A sync cycle has two visible phases:

1. **Checking** — fetching remote state (branch info, snapshot). No mutations yet. The spinner shows:

```
◐ checking...
```

2. **Syncing** — mutations are being applied. The spinner transitions to show each file with a direction arrow:

```
◐ syncing... ↑ hello.md
◐ syncing... ↓ photo.jpg
◐ syncing... ↑↓ notes/todo.md
```

If there are no changes, the spinner goes straight from "checking..." to the final line — "syncing..." is never shown.

Direction arrows map from `FileMutation`:
- **↑** pushed to remote — `disk: "skip"` with `remote: "write"|"delete"`
- **↓** pulled from remote — `remote: "skip"` with `disk: "write"|"delete"`
- **↑↓** both (merged) — `disk: "write", remote: "write"`

`disk: "skip", remote: "skip"` mutations (no-ops) are omitted from display.

**Color (TTY only):**
- Header/muted text: dim (gray)
- Checking/syncing spinner: yellow
- Success (`✓`): green
- Error (`✗`): red
- Arrows and file paths: default

**Non-TTY:** no spinner, no overwriting, no color. One summary line per sync.

### `stash sync`

The sync command uses the shared sync output, then prints a final line and exits:

```
$ stash sync
◐ checking...
◐ syncing... ↑ hello.md
◐ syncing... ↓ photo.jpg
✓ synced (2↑ 1↓)
```

No changes:
```
$ stash sync
◐ checking...
✓ up to date
```

Error:
```
$ stash sync
✗ sync failed: network error
```

```
$ stash sync
✗ sync failed: sync already in progress
```

### `stash watch`

Watches the current directory. Runs in the foreground until interrupted. Requires a connection — if none is configured, exits immediately:

```
no connection configured — run `stash connect <provider>` first
```

#### Sync triggers

A sync is triggered by two things:

1. **Filesystem changes** — files created, edited, or deleted in the stash directory. Debounced: the watcher waits 1 second of quiet after the last change before syncing. Dotfiles, symlinks, and `.stash/` are ignored (same rules as `scan()`).

2. **Periodic poll** — every 30 seconds, a sync runs regardless of local changes. This picks up remote changes (no push notification from GitHub).

After any sync completes, the poll timer resets to a full 30 seconds. If a poll fires during a debounce window, the poll is skipped — the debounced sync covers both local and remote changes.

If filesystem events arrive while a sync is in progress, they are not lost. When the sync finishes, a new debounce cycle starts from the queued events.

An initial sync runs immediately on startup.

#### Output

A static header prints once on startup (dim/muted):

```
watching /Users/me/notes (. to sync, q to quit)
```

Below it, a single live-updating status line. During sync, it uses the shared sync output (checking → syncing). Otherwise:

**Idle after sync with changes:**
```
● 2↑ 1↓ 1↑↓ · checking in 27s
```

**Idle after check with no changes:**
```
● up to date · checking in 14s
```

**Checking/syncing:** shared sync output (spinner + phases).

**Error:**
```
✗ sync failed: network error · retrying in 27s
```

The countdown ticks down live so the user knows when the next check fires. On first launch, the watcher goes straight to `◐ checking...` with no idle state. Errors are shown but the watcher keeps running. Retries every 30 seconds, no backoff.

**Non-TTY:** one-line-per-sync log output. No spinner, no overwriting, no color, no raw mode keyboard input.

#### Keyboard

Stdin is in raw mode (TTY only):

- **`.`** — trigger an immediate sync. Debounce still applies if files are actively changing.
- **`q`** or **Ctrl-C** — graceful shutdown.

#### Shutdown

On `q` or Ctrl-C:

1. Stop the filesystem watcher and poll timer.
2. If a sync is in progress, wait for it to finish.
3. Restore terminal from raw mode.
4. Print: `stopped watching /path/to/stash`
5. Exit 0.

A second Ctrl-C during the wait force-exits immediately.

### Cross-process safety

A file lock (`.stash/sync.lock`) prevents concurrent syncs across processes. `sync()` acquires the lock before running and releases it in `finally`. If the lock is held, `sync()` throws `SyncLockError`. Callers handle it like any other error.

Lock details:
- Lock file: `.stash/sync.lock`
- Acquired via atomic file create (`wx` flag — fails if file already exists)
- Contains JSON: `{ "pid": 12345, "startedAt": "2026-03-03T12:00:00Z", "hostname": "machine-a" }`
- Stale lock TTL: 10 minutes — if `startedAt` is older than 10 minutes, assume crashed process and reclaim (remove + retry once)
- Released in `finally` (delete the file)

## Implementation

### Dependencies

- **`@parcel/watcher`** — native filesystem watcher (FSEvents on macOS, inotify on Linux). Batched events, low overhead.
- **`@rupertsworld/emitter`** — typed event emitter with disposable subscriptions. Replace vendored `src/emitter.ts` with package import.
- **`@rupertsworld/disposable`** — `Disposable` interface and `DisposableGroup` for resource cleanup.

### Color

Use `src/ui/color.ts` — a tiny TTY-aware color module (no dependency). Exports functions that wrap text in ANSI codes only when the stream is a TTY:

```ts
function createColors(stream: NodeJS.WriteStream): {
  dim: (s: string) => string
  yellow: (s: string) => string
  green: (s: string) => string
  red: (s: string) => string
}
```

If `stream.isTTY` is false, all functions return the input unchanged. Colors are wired through `LiveLine` → `SyncRenderer`.

### Module structure

```
src/
├── ui/
│   ├── color.ts            # TTY-aware ANSI color helpers
│   ├── live-line.ts        # terminal line primitive + spinner
│   ├── format.ts           # pure: arrows, summaries, time-ago, countdown
│   └── sync-renderer.ts    # mutation events → display
├── watch.ts                # watch orchestration (CLI layer)
└── ...
```

Watch orchestration lives in `src/watch.ts`, not in Stash core. Stash exposes `sync()` and mutation events — it knows nothing about watching, timers, or terminal output.

### UI interfaces

#### `format.ts`

Pure functions, no state, no I/O:

```ts
type Direction = "up" | "down" | "both"

function mutationDirection(m: FileMutation): Direction
function directionArrow(d: Direction): string        // "↑", "↓", "↑↓"
function formatSummary(mutations: FileMutation[]): string   // "2↑ 1↓ 1↑↓" or ""
function formatTimeAgo(date: Date): string           // "just now", "3s ago", "2m ago"
function formatCountdown(targetDate: Date): string   // "27s", "1m", "0s"
```

#### `live-line.ts`

```ts
class LiveLine implements Disposable {
  constructor(stream: NodeJS.WriteStream)

  update(text: string): void        // overwrite current line (TTY only)
  print(text: string): void         // permanent line (TTY and non-TTY)
  startSpinner(text: string): void  // spinner frames prepended to text
  spinnerText(text: string): void   // update text while spinner is active
  stopSpinner(): void
  dispose(): void
}
```

TTY: `update()` overwrites with `\r` + clear. Non-TTY: `update()` is a no-op, `print()` writes a line. Spinner frames: `["◐", "◓", "◑", "◒"]` at ~80ms.

#### `sync-renderer.ts`

Created fresh per sync cycle. Accumulates mutations for the summary.

```ts
class SyncRenderer implements Disposable {
  constructor(line: LiveLine)

  onMutation(mutation: FileMutation): void  // updates spinner line with arrow + path
  done(): string                            // stops spinner, returns summary ("2↑ 1↓" or "")
  error(err: Error): void                   // stops spinner
  dispose(): void
}
```

Both `stash sync` and `stash watch` wire `stash.on("mutation")` → `renderer.onMutation()`, call `done()` on success, and use the returned summary to format the final line.

### Watch state machine

Three states, local to the `watch()` function:

```ts
type WatchState = "idle" | "debouncing" | "syncing"
```

| From | Event | To | Action |
|------|-------|----|--------|
| `idle` | fs event | `debouncing` | start debounce timer (1s) |
| `idle` | poll timer fires | `syncing` | run sync |
| `idle` | `.` keypress | `syncing` | run sync |
| `debouncing` | fs event | `debouncing` | reset debounce timer |
| `debouncing` | debounce timer fires | `syncing` | run sync |
| `debouncing` | poll timer fires | `debouncing` | skip poll |
| `debouncing` | `.` keypress | `syncing` | cancel debounce, run sync |
| `syncing` | fs event | `syncing` | set `pendingEvents` flag |
| `syncing` | poll timer fires | `syncing` | skip poll |
| `syncing` → done | `pendingEvents` set | `debouncing` | start debounce timer |
| `syncing` → done | no pending events | `idle` | reset poll timer, update status line |
| `syncing` → error | — | `idle` | show error, reset poll timer |

All resources (watcher subscription, timers, stdin listener) tracked in a `DisposableGroup`.

### Sync lock

The in-process `syncInFlight` guard becomes the first check. The file lock is the cross-process guard. Both throw `SyncLockError`.

```ts
class SyncLockError extends Error {}
```

Add to `src/errors.ts` alongside `PushConflictError`.

Acquire order in `sync()`: check `syncInFlight` → acquire file lock → set `syncInFlight` → run sync → release in `finally` (clear flag + delete file).

Stale lock reclaim: read lock file, parse `startedAt`, if older than 10 minutes → delete and retry `wx` create once. If second create also fails → throw `SyncLockError`.

### Path filtering

Extract the dotfile/symlink/`.stash/` predicate from `listTrackedFiles()` into a shared utility so both `scan()` and the `@parcel/watcher` event filter use identical rules.

## Tests

### Target: `spec/stash.md` — unit tests

#### sync lock

```
1. Lock acquired and released on successful sync
   - makeStash with files, connect, FakeProvider
   - sync()
   - .stash/sync.lock does not exist after sync completes

2. Lock released on sync failure
   - makeStash, connect, FakeProvider that throws on push
   - sync() → throws
   - .stash/sync.lock does not exist

3. SyncLockError when lock held by another process
   - makeStash, connect
   - Manually write .stash/sync.lock with recent startedAt
   - sync() → throws SyncLockError

4. Stale lock is reclaimed
   - makeStash, connect, FakeProvider
   - Write .stash/sync.lock with startedAt 15 minutes ago
   - sync() → succeeds (stale lock reclaimed)
   - .stash/sync.lock does not exist after sync

5. Stale lock reclaim race
   - makeStash, connect
   - Write .stash/sync.lock with startedAt 15 minutes ago
   - Delete lock and immediately re-create with recent startedAt (simulate race)
   - sync() → throws SyncLockError

6. In-process single-flight throws SyncLockError
   - Start sync() with a slow FakeProvider
   - Call sync() again concurrently
   - Second call throws SyncLockError
   - First completes normally
```

#### isTrackedPath()

```
1. Regular file path → true
2. Dotfile (.hidden) → false
3. Dot-directory (.config/settings.json) → false
4. .stash/ path → false
5. Nested regular path (a/b/c.md) → true
```
